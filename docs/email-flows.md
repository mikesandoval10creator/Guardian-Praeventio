# Email flows (Resend)

Sprint 22 Bucket Y. Centralizes every transactional email Praeventio
emits via Resend, with a single service (`src/services/email/`) so
templates, From identity, and audit hooks stay consistent.

## Configured flows

| Flow | Trigger | Recipients | Template | Tag |
|------|---------|-----------|----------|-----|
| SOS backup | `POST /api/emergency/sos` when push fails or no FCM tokens | Project supervisors / gerentes / prevencionistas | `sosBackupTemplate` | `sos-backup` |
| Project invitation | `POST /api/projects/:id/invite` | Invited email | `projectInvitationTemplate` | `invitation` |
| Weekly digest | Cron Mon 09:00 Santiago via `POST /api/admin/jobs/weekly-digest` | Supervisors per active project | `weeklyDigestTemplate` | `weekly-digest` |
| Calendar invite | Calendar event creation | Attendees | `calendarInviteTemplate` + `.ics` attachment | `calendar-invite` |
| Incident alert | High/critical incident creation | Project supervisors | `incidentAlertTemplate` | `incident-alert` |
| Password reset | Firebase Auth (handled outside this service) | n/a | n/a | n/a |

The audit row footer in every template carries the originating
`audit_logs` ID and a `reason` line so a SUSESO auditor can match an
inbox copy to the originating Firestore write.

## Configuration

### Required env vars

- `RESEND_API_KEY` — Resend API key. Provisioned in Secret Manager.
  When missing, `EmailService.fromEnv()` returns `null` and call sites
  silently no-op (the underlying domain action — SOS write, invite
  doc, digest log — still completes; only the email side-effect is
  skipped).
- `RESEND_FROM_ADDRESS` — Optional override of the From header.
  Defaults to `Praeventio <noreply@praeventio.app>`.
- `APP_URL` — Public app URL used in template CTAs and deep-links.
  Defaults to `https://app.praeventio.app`.

### DNS records (verified domain)

For `praeventio.app` (or whichever domain is in `RESEND_FROM_ADDRESS`):

- **SPF (TXT @ ):** `v=spf1 include:amazonses.com ~all` (Resend uses
  Amazon SES on the back end).
- **DKIM:** Three CNAME records as shown in the Resend dashboard
  (`resend._domainkey`, `resend2._domainkey`, `resend3._domainkey`)
  pointing at `*.dkim.amazonses.com`.
- **DMARC (TXT _dmarc):** `v=DMARC1; p=quarantine; rua=mailto:dmarc@praeventio.app; pct=100; adkim=s; aspf=s`
  — start at `p=none` for a week to inspect aggregate reports, then
  promote to `quarantine` once aligned.
- **MX (optional):** Only required if you want to receive email at the
  domain; transactional sends do not need it.

Verify in the Resend dashboard before flipping production traffic over.

## Templates

All templates are HTML strings produced by pure functions in
`src/services/email/templates.ts`. Layout primitives:

- Brand teal `#4db6ac` for CTAs and accents.
- Petroleum `#014c66` for headings and the header bar.
- Coral `#ff6f61` reserved for SOS / critical incidents.
- Single-table layout (no flex / grid) for Outlook compatibility.
- Plain-text fallback auto-generated by `EmailService` from the HTML.
- `escapeHtml` helper applied to every payload field — never embed
  unsanitized user input.

### Customizing a template

Edit the function in `src/services/email/templates.ts`. The exported
helpers `header()`, `footer()`, and `shell()` keep brand consistency;
modify the body block of the specific template only.

When adding a new template:

1. Add the function + payload interface to `templates.ts`.
2. Re-export from `src/services/email/index.ts`.
3. Add a snapshot/contains test in `resendService.test.ts`.
4. Document the new flow in this file with its tag.

## Audit + frequency

| Flow | Frequency | Audit log action |
|------|-----------|------------------|
| SOS backup | On-demand (per SOS call) | `emergency.sos` (parent row references `alertId`) |
| Project invitation | On-demand (per invite) | implicit via the invitations Firestore doc |
| Weekly digest | Mon 09:00 Santiago | `weekly_digest_run` |
| Calendar invite | On-demand | calendar event doc itself |
| Incident alert | On-demand (severity ≥ high) | `incident.created` |

The `weekly_digest_run` audit row stores the aggregated counters
(`projectsProcessed`, `projectsSent`, `totalEmailsSent`,
`totalEmailErrors`) so ops can spot a regression without reading Cloud
Run logs.

## Failure semantics

`EmailService.send` and `sendBatch` never throw — they return a
discriminated `{ ok, error }` result. Call sites must:

- Treat email as **best-effort**. The primary domain action (SOS
  write, invitation doc, weekly stat aggregation) must succeed
  regardless of email outcome.
- Log partial-failure batches at `warn` level (see SOS handler).
- Avoid retrying inside the request path — Resend already has its own
  retry; a server-side retry loop just amplifies rate-limit pressure.

## Local development

Without `RESEND_API_KEY` set, the service no-ops cleanly:

```bash
unset RESEND_API_KEY
npm run dev
# POST /api/emergency/sos succeeds, returns { ok: true, alertId, notified, emailedSupervisors: 0 }
```

To exercise the templates without sending, render them in a Vitest
test (see `src/services/email/resendService.test.ts`) or paste the
output of any `*Template(payload)` into a browser tab.
