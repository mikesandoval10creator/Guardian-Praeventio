2.1 QuÈ es Guardian-Praeventio

Guardian-Praeventio es una plataforma safety-critical orientada al mercado chileno y LATAM que cubre
prevenciÛn de riesgos, cumplimiento normativo (SUSESO, DS 67, DS 76, DS 109, Ley 21.719), reporte de
incidentes, gestiÛn de EPP, evacuaciones, Arbeitssicherheit, higiene industrial, ergonomÌa
(RULA/REBA/TMERT/IPER/PreXoR/PlaneSI), fatiga, lone-worker, brigadas de emergencia, DEA,
simulaciÛn de contingencia, y un largo etcÈtera de dominios de seguridad ocupacional. El stack tecnolÛgico es
complejo: frontend React 19 + Vite 6 + TypeScript 5.8 + Tailwind 4 + React Router 7 con lazy chunks;
backend Express 4 monolÌtico con 210 routers montados en server.ts (1.552 LOC); persistencia en Firestore
con reglas default-deny de 1.829 LOC; autenticaciÛn Firebase Auth + WebAuthn + TOTP; pagos vÌa Webpay,
MercadoPago, Khipu, Google Play Billing y Apple App Store; despliegue en Google Cloud Run (regiÛn
southamerica-west1 para data residency Ley 19.628); y empaquetado mobile vÌa Capacitor 8 con 13 plugins
nativos (biometric-auth, bluetooth-le, sqlite, geolocation, motion, push-notifications, foreground-service,
proximity, health-connect, healthkit, etc.).

La condiciÛn de safety-critical cambia la barra de calidad: las features de vida-safety (SOS, ManDown,
lone-worker, evacuation, brigade, DEA, incident reporting) son FREE en todos los tiers incluyendo free, seg˙n
la convenciÛn #11 de CLAUDE.md, y cualquier regresiÛn en estos flujos puede tener consecuencias fÌsicas para
los trabajadores en faena. Esto hace que la cobertura e2e real de estos flujos sea un requisito de producto, no
un nice-to-have de ingenierÌa.

8.7 Brecha entre CI y local: harness gap

El harness gap que mantiene los 3 specs vida-safety en fixme es parcialmente atribuible a la diferencia entre el
entorno CI y el entorno local de desarrollo. EspecÌficamente:

 ∑ Java version: CI usa actions/setup-java@temurin-17; local requiere Temurin 21 seg˙n los
     comentarios en cada spec fixme. Es posible que Firestore Emulator tenga comportamiento ligeramente
     distinto entre Java 17 y 21.

 ∑ Build mode: CI ejecuta npm run build -- --mode test; local suele usar npm run test:e2e:full
     que NO hace el build con --mode test. El script npm deberÌa incluir el build para reproducir CI
     exactamente.

 ∑ FIREBASE_AUTH_EMULATOR_HOST: CI lo setea explÌcitamente a 127.0.0.1:9099; el script
     npm test:e2e:full no lo incluye, lo que puede causar que el fixture auth.ts no encuentre el Auth
     Emulator.

 ∑ Workers: CI usa workers: 1 (serialized); local suele usar default (auto, paralelo). Tests que dependen de
     estado Firestore compartido pueden pasar en paralelo pero fallar en serie o viceversa.

La Fase 1.1 del roadmap es diagnosticar y cerrar este harness gap. La soluciÛn m·s directa es agregar un script
test:e2e:ci que reproduzca exactamente los pasos del job CI, incluyendo build --mode test, JAVA_HOME
apuntando a Temurin 21, FIREBASE_AUTH_EMULATOR_HOST explÌcito, y workers: 1.
